# T22 — App-two mobile shell: Lusail branding + guest Home (news) & Matches (fixtures/results/table)

**Branch:** `task/T22-app-two-shell` (cut from current `main` = T21 `bc79df3`; verify base with `git merge-base --is-ancestor main HEAD`). One task, one branch. **Commit everything before handoff** (clean worktree). **Do NOT run `flutter create`** (it re-adds desktop/web platforms + a default `widget_test.dart` that breaks `flutter test`). Reference toolchain: **Flutter 3.44.4**. Mobile-only — **no backend/PHP changes** (`php artisan test` stays green & untouched).

**This is the first app-two mobile task.** It rebrands `demo_app_two` to the Lusail SC identity and builds a **guest-first** shell whose **Home** (club news) and **Matches** (fixtures / results / league table) tabs read the **public T20 `/content/*` API with no login**. Member auth + wallet is T23; VVIP card is T24 — this task only scaffolds the shell + the two guest tabs (and a gated third tab that prompts sign-in).

## Product decisions (locked with the user)
- **Guest nav:** show the **full** bottom nav to guests (Home, Matches, Membership). Tapping the gated **Membership** tab shows a "Sign in to access" prompt with a login button (the button can be a stub in T22 — T23 wires the real login).
- **Role routing:** **one authenticated shell**, role-driven — the third tab will render discount-wallet (member) vs VVIP-card (vvip_member) from `account_type` in `/me`, later (T23/T24). For T22, authed users just see a simple placeholder on that tab.

## Architecture — mirror how app one is wired
`CoreApp` (app one's *entire* shell) lives in **`lfc_core`**; `demo_app_one` only injects its `Brand` + base URL and renders `CoreApp`. Follow the same pattern: build a **sibling root `SupporterApp` in `lfc_core`**, exported from the barrel, and have `demo_app_two` render it. App-two feature code (content models/repos + screens) lives in `lfc_core/lib/src/` alongside app one's features (so it shares theme, l10n, Dio, session). Keep `CoreApp` and every existing app-one file **untouched** — this is additive.

Optional polish: set the native display label to "Lusail SC" (Android `android:label`, iOS `CFBundleDisplayName`) — T16 left it "Demo Two".

### 2. `AppConfig` carries an app key → Dio sends `X-App-Key`
The guest `/content/*` endpoints scope by the `X-App-Key` header (T20). Add a nullable `appKey` to `AppConfig` (`lib/src/config/app_config.dart`), threaded through `AppConfig.fromEnvironment({defaultBaseUrl, appKey})`. In `dioProvider` (`lib/src/providers.dart`), if `config.appKey != null`, add `options.headers['X-App-Key'] = config.appKey` in the `onRequest` interceptor (alongside the existing Accept/Accept-Language). **App-one safety:** `demo_app_one` doesn't set `appKey` → header absent → app one unchanged; authed endpoints ignore the header anyway. `demo_app_two`'s `app.dart` overrides `appConfigProvider` with `appKey: 'app_two'`.

### 3. Content feature in `lfc_core` (`lib/src/features/content/`)
- **Models** (plain immutable classes, `fromJson`): `NewsSummary` (id, title, excerpt, imageUrl, publishedAt), `NewsArticle` (+ body), `MatchSummary` (id, opponent, competition, isHome, venue, kickoffAt, status, ourScore, opponentScore — scores nullable), `StandingRow` (position, clubName, played, won, drawn, lost, goalsFor, goalsAgainst, goalDifference, points, isOwnClub).
- **`ContentRepository`** (constructed from `dioProvider`) with `fetchNews()`, `fetchNewsArticle(int id)`, `fetchFixtures()`, `fetchResults()`, `fetchStandings()` → hit `/content/news`, `/content/news/{id}`, `/content/fixtures`, `/content/results`, `/content/standings`, unwrap the `{data: ...}` envelope, map JSON→models. No auth needed (guest).
- Riverpod providers in `providers.dart`: `contentRepositoryProvider`, plus `newsProvider`/`fixturesProvider`/`resultsProvider`/`standingsProvider` as `FutureProvider.autoDispose` (mirror `playersProvider`). Server already localises text via `Accept-Language` (the Dio sends it), so screens just display returned strings.

### 4. `SupporterApp` root + guest-first shell (`lib/src/`)
- **`SupporterApp`** (e.g. `lib/src/supporter_app.dart`, exported from the barrel): a `MaterialApp` mirroring `CoreApp`'s setup (title=`brand.appTitle`, `buildLightTheme/DarkTheme(brand.palette)`, `themeMode`, `locale`, `AppLocalizations` delegates + `supportedLocales`) but with `home:` = the supporter shell. Keep it standalone (don't refactor `CoreApp`, to leave app one untouched).
- **`SupporterShell`** — a `Scaffold` with a `BrandAppBar` (reuse `theme/widgets/brand_app_bar.dart` — includes the theme + language toggles) and a `NavigationBar` (bottom) with three destinations: **Home**, **Matches**, **Membership** (icons + localised labels). Body via `IndexedStack`.
  - **Home tab** → `HomeNewsScreen`: a broadcast-style list of `newsProvider` (hero image if `imageUrl != null`, title, excerpt, relative/formatted `publishedAt`); tapping opens `NewsDetailScreen` (`fetchNewsArticle` → title, image, body). Loading/empty/error states (reuse the app-one pattern: `.when(data/loading/error)`; friendly error via `ApiException.toString()`).
  - **Matches tab** → `MatchesScreen`: a `TabBar`/segmented control with three views — **Fixtures** (`fixturesProvider`, upcoming, each row: opponent, competition, home/away chip, formatted kickoff), **Results** (`resultsProvider`, each row: `Lusail  {ourScore}–{opponentScore}  {opponent}`, competition, date), **Table** (`standingsProvider`, a scrollable standings table: position, club, P/W/D/L/GD/Pts columns; highlight the `isOwnClub` row in brand gold/navy).
  - **Membership tab** → `MembershipGateScreen`: watch `sessionControllerProvider` — if **unauthenticated**, show a centered "Sign in to access your membership" card + a "Sign in" button (stub `onPressed` for now — a `// TODO(T23)` comment; do NOT build the login flow here); if **authenticated**, show a simple placeholder ("Your membership — coming soon") — T23/T24 replace this.
- Wide content (the standings table) must live in a horizontally-scrollable container so the page body never scrolls sideways; keep controls reachable in the 600px test viewport (see the mobile-design-system notes on the 800×600 test viewport + finders skipping off-stage widgets).
