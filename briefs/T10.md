## Part 1 — Fold in three fixes

### Fix A — platform-aware API base URL default
**File:** `mobile/lib/src/config/app_config.dart`

Today it hard-defaults to `http://localhost:8000/api/v1`, which cannot work on the **Android emulator** (there, `localhost` = the emulated device, not the host — the host is reachable only via `10.0.2.2`). Android Studio's Run button doesn't pass `--dart-define`, so it silently uses the broken default.

Change `AppConfig.fromEnvironment()` to:
- Honor `--dart-define=API_BASE_URL=...` when provided (unchanged behavior).
- Otherwise default to **`http://10.0.2.2:8000/api/v1` on Android** and **`http://localhost:8000/api/v1` elsewhere** (iOS simulator / desktop). Use `dart:io Platform.isAndroid`.

Sketch:
```dart
import 'dart:io' show Platform;

factory AppConfig.fromEnvironment() {
  const override = String.fromEnvironment('API_BASE_URL');
  if (override.isNotEmpty) return AppConfig(apiBaseUrl: override);
  final host = Platform.isAndroid ? '10.0.2.2' : 'localhost';
  return AppConfig(apiBaseUrl: 'http://$host:8000/api/v1');
}
```
Update the login screen's `demoApiHint` string (EN **and** AR) to reflect the new auto-default. The test `appConfigProvider` override keeps working (tests override the provider value, so `Platform` is never hit in tests).

> For a **physical Android device** on the same Wi-Fi, the demo runner still passes `--dart-define=API_BASE_URL=http://<mac-lan-ip>:8000/api/v1` and serves with `php artisan serve --host=0.0.0.0`. Document this in the walkthrough (Part 3).

### Fix B — `ApiException.toString()`
**File:** `mobile/lib/src/core/api/api_exception.dart`

Six screens (offers, rewards, players, points-history, staff-scanner, qr) render load failures via `error.toString()`, which currently yields the useless `Instance of 'ApiException'`. Add:
```dart
@override
String toString() => message;
```
This repairs all six at once (they show the friendly `message`, e.g. "Cannot reach the server. Check the API URL."). Add a one-line test asserting `ApiException(message: 'x', kind: ApiErrorKind.unknown).toString() == 'x'`.

### Fix C — balances go stale after a redeem (and after a scan)
Two surfaces don't refresh after points change:
- **Players tab** (`mobile/lib/src/features/players/presentation/players_screen.dart`): loads `points_balance` once via `late Future _future = fetchPlayers()` in `initState`, and it lives inside an **`IndexedStack`** (`mobile/lib/src/features/home/presentation/home_shell.dart:131`) that keeps it alive — so it never re-fetches after a redeem in the Rewards tab.
- **Profile header** (`home_shell.dart:121`): shows `account.accountBalance` from `session.account`, a login-time snapshot that is never refreshed.

**Fix:**
1. Expose the players list as a **shared, invalidatable provider** — e.g. `playersProvider = FutureProvider.autoDispose((ref) => ref.watch(playerRepositoryProvider).fetchPlayers())` — and have `PlayersScreen` `ref.watch` it instead of a private `Future`.
2. Add `SessionController.refreshAccount()` that re-fetches `/me` and updates `state.account`.
3. After a successful redeem in `rewards_screen._redeem()`, call **`ref.invalidate(playersProvider)`** *and* **`refreshAccount()`** (in addition to the existing local `_refresh()`).
4. For demo robustness, also `ref.invalidate(playersProvider)` after a successful **staff scan** credit, and keep/confirm pull-to-refresh on the Players tab.
5. Add a widget test: fake repo returns a lower balance on the second `fetchPlayers()`; assert the Players list reflects the decrement after a redeem.

> **Correctness note:** a plain parent's *account* balance chip legitimately does **not** change on a player redeem (different ledger — the points live on the player). Don't "fix" that by subtracting from the header. The parent's visible change is the **per-player balance** on the Players tab; the header chip only changes for **VVIP account-path** redeems.

## Part 2 — Seed data guarantees (enrich existing seeders; do NOT rewrite)

The seeders already produce a coherent demo: admin (`admin@lfc.test`), season, 2 teams (LFC U12 / U14), 2 fixtures (Al Sadd = `OpenForScanning`, Al Rayyan = `Scheduled`), 2 point rules (fixed 10 pts + team 5%-of-fee), redemption items, offers, demo parent `parent.demo@lfc.test` → player **Omar Demo** (150 starting pts, on LFC U12), and VVIP client `vvip.demo@lfc.test` (Sheikha, 500 pts, no players).

**Verified current state (as of this brief, on a fresh `migrate:fresh --seed`):** Omar is on LFC U12; the open "Al Sadd SC" fixture is also LFC U12; `isOpenForScanning()` is true — so the scan currently credits Omar correctly. The job below is to make that **guaranteed and robust**, not accidental, and to add an integration test that fails loudly if any invariant breaks.

Make these invariants guaranteed, all idempotent (`updateOrCreate`/`firstOrCreate`), and assert each in a new seeder-invariants test:

1. **Demo player is on the open fixture's team.** Currently `FixtureSeeder` resolves the team as `LFC U12 ?? first` while `ParentAccountSeeder` uses `Team::first()` — they align only because U12 is first. Make it explicit: resolve both to the **same** team (link Omar to the open fixture's team, or target `LFC U12` by name in both). If they diverge, scanning credits nobody (422). **Assert `demoPlayer.team_id === openFixture.team_id`.**
2. **Open fixture stays scannable even if the DB was seeded hours earlier.** Widen the window: `scan_opens_at = now()->subDay()`, `scan_closes_at = now()->addDays(7)`, status `OpenForScanning`. **Assert `isOpenForScanning()` is true.**
3. **Offers: ≥1 `all`-audience and ≥1 `vvip`-audience offer, both valid now.** **Assert** the parent sees only `all` via `Offer::visibleTo($parent)` and the VVIP sees both. (Check `OfferSeeder` validity windows actually cover `now`.)
4. **Redeemable balances:** ≥1 redemption item `in_stock` and costing ≤ 150 (Omar can redeem) and ≤ 500 (VVIP). **Assert** an affordable in-stock item exists.
5. **(Nice-to-have; keeps the admin dashboard non-empty)** seed one already-`fulfilled` redemption and one `issued` (pending) redemption so the loyalty dashboard + "Pending fulfillments" widget show real rows. Idempotent.

`migrate:fresh --seed` must remain the single reset command. Keep `php artisan test` green.
