**This is mobile-only. No backend/API/PHP changes** — `php artisan test` must stay green and untouched. The app's *runtime behavior* (screens, auth, RTL, features) must be **byte-for-byte identical** after the refactor for `demo_app_one` (it IS today's app, reassembled).

## The seam — what is shared vs. injected

Only **two** things differ per demo: **branding** and **API base URL**. Everything else (auth, offers, players, redemptions, scan, session, staff, api, storage, providers, locale, theme *engine*, l10n) is shared in `lfc_core`.

Inject both via **Riverpod `ProviderScope` overrides** — this reuses the pattern already in the code (`appConfigProvider` already exists in `providers.dart:25`).

**1. `appConfigProvider` (runtime config).** Already `Provider<AppConfig>` → `AppConfig.fromEnvironment()`. Extend `AppConfig.fromEnvironment` to accept a per-app **default** base URL while still honoring `--dart-define=API_BASE_URL` and the platform default (Android `10.0.2.2` / iOS `localhost`) from T10. Each app overrides `appConfigProvider` with its own default.

**2. `brandProvider` (new, in `lfc_core`).** Introduce a `Brand` value object + a provider that has **no default** (throws if not overridden), forcing each app to supply one:
```dart
// lfc_core: lib/src/branding/brand.dart
@immutable
class Brand {
  const Brand({required this.appTitle, required this.palette, required this.logo});
  final String appTitle;         // -> MaterialApp.title
  final BrandPalette palette;    // seed colours (navy primary, gold, hero gradient) — see theme note
  final ImageProvider logo;      // -> brand_mark widget
}
final brandProvider = Provider<Brand>(
  (ref) => throw UnimplementedError('override brandProvider per app'),
);
```

**3. `CoreApp` (rename of `LfcApp`, `src/app.dart`).** Today it hardcodes `title: 'LFC'`, `buildLightTheme()`, `buildDarkTheme()`. Change it to read `final brand = ref.watch(brandProvider);` and use `brand.appTitle` + `buildLightTheme(brand.palette)` / `buildDarkTheme(brand.palette)`. Routing/session logic unchanged.

**4. Theme (`src/theme/app_theme.dart`).** Today `LfcColors` are hardcoded consts feeding `buildLightTheme()/buildDarkTheme()`. Refactor `buildLightTheme`/`buildDarkTheme` to take a `BrandPalette` (the handful of seed colours: crest navy `navy700`, the navy scale or a way to derive it, `gold`/`goldBright`, hero gradient, success). **Provide a default `BrandPalette` equal to today's exact Lusail values** so `demo_app_one` renders identically. Keep `LfcPalette`/`ThemeExtension` mechanics; just source their inputs from the passed `BrandPalette` instead of top-level consts. (Keep it pragmatic — parameterise the ~6 brand-defining colours, not every neutral.)

**5. `brand_mark.dart` (`src/theme/widgets/`).** Swap its hardcoded logo asset for `ref.watch(brandProvider).logo`.

**Public barrel** `lfc_core/lib/lfc_core.dart` should export `CoreApp`, `brandProvider`, `Brand`, `BrandPalette`, `appConfigProvider`, `AppConfig` so app `main.dart` files import only `package:lfc_core/lfc_core.dart`.

**Each app `main.dart` becomes ~10 lines**, e.g. `demo_app_one`:
```dart
import 'package:flutter/widgets.dart';
import 'package:flutter_riverpod/flutter_riverpod.dart';
import 'package:lfc_core/lfc_core.dart';
import 'brand.dart';   // demoOneBrand
import 'config.dart';  // demoOneApiBaseUrl

void main() {
  WidgetsFlutterBinding.ensureInitialized();
  runApp(ProviderScope(
    overrides: [
      brandProvider.overrideWithValue(demoOneBrand),
      appConfigProvider.overrideWith(
        (ref) => AppConfig.fromEnvironment(defaultBaseUrl: demoOneApiBaseUrl),
      ),
    ],
    child: const CoreApp(),
  ));
}
```

## Assets & fonts (known-fiddly — verify empirically)

- **Per-app logo** (today `assets/images/lusail_crest.png`, `lusail_logo.png`) is *branding* → lives in each **app** package (`apps/<x>/assets/images/`) and is delivered via `Brand.logo`. demo_app_one keeps the Lusail images; demo_two/three get placeholders.
- **Shared assets** — the **fonts** (Changa, Tajawal) and **flags** (`gb.png`, `qa.png`, used by the language toggle) move into `packages/lfc_core/assets/` (generic, identical across demos).
- **Referencing shared assets from core code:** images via `AssetImage('assets/flags/qa.png', package: 'lfc_core')`.
- **Fonts are the tricky part.** Core's theme sets bare `fontFamily: 'Changa'`/`'Tajawal'`. Recommended pattern: keep the font *files* under `packages/lfc_core/assets/fonts/`, and **declare the font families in each app's pubspec** pointing at the package path so the bare family names resolve app-wide:
  ```yaml
  # apps/demo_app_one/pubspec.yaml
  flutter:
    fonts:
      - family: Changa
        fonts: [{ asset: packages/lfc_core/assets/fonts/Changa-VF.ttf }]
      - family: Tajawal
        fonts:
          - asset: packages/lfc_core/assets/fonts/Tajawal-Regular.ttf
          - { asset: packages/lfc_core/assets/fonts/Tajawal-Medium.ttf, weight: 500 }
          - { asset: packages/lfc_core/assets/fonts/Tajawal-Bold.ttf, weight: 700 }
  ```
  (Also declare the fonts/flags under `lfc_core`'s own `flutter:` section so the package bundles them.) **If** the package-qualified font path doesn't resolve cleanly, the accepted fallback is to duplicate the font files into each app package. **Acceptance is empirical:** the running app must show Changa headings + Tajawal body, not the fallback font — verify on device, in both EN and AR.
- **l10n:** move `lib/l10n/*` + `l10n.yaml` into `lfc_core`; keep `generate: true` in `lfc_core/pubspec.yaml`; run `flutter gen-l10n` there. Apps don't touch localization delegates (the `MaterialApp` is built inside `CoreApp`).
