## Why

## Part 1 â€” Backend: app-scope the staff fixtures picker

**The trap:** `SetAppContextFromHeader` falls back to **`AppKey::AppTwo`** when the header is missing/invalid (correct for the guest `/content/*` routes â€” only app two calls them). But **app one sends no `X-App-Key` at all** (`demo_app_one`'s `AppConfig.appKey` is null; the Dio interceptors in `providers.dart` only attach the header when `appKey` is non-empty). So simply adding the existing middleware to `/staff/fixtures` would hand app one **app-two** fixtures â€” inverting the bug, not fixing it.

**Fix it belt-and-braces:**

1. **Parameterize the middleware default.** `SetAppContextFromHeader::handle(Request $request, Closure $next, ?string $default = null)`: resolve `$fallback = AppKey::tryFrom((string) $default) ?? AppKey::AppTwo`, then `$app = AppKey::tryFrom((string) $request->header('X-App-Key')) ?? $fallback`. Existing (unparameterized) usage on the `/content/*` group keeps its **AppTwo** default â€” **unchanged**. Keep the `finally { $context->clear(); }`.
2. **Apply it to the staff picker only** â€” `Route::get('/staff/fixtures', â€¦)->middleware(SetAppContextFromHeader::class.':app_one')`. So a header-less legacy client still sees exactly today's app-one fixtures. **Do NOT** apply it to the whole `auth:sanctum` group: that would put every authed route under the global scope and change `/me`, `/players`, `/offers` (which already scopes itself via `forApp($parent->app)`, T21) â€” needless blast radius.
3. **Do NOT put app context on `POST /scan`.** The global scope would then filter both `Fixture::findOrFail($data['fixture_id'])` **and** `ParentAccount::findOrFail($parentId)`, which breaks scanning across builds and adds a hard failure mode for no gain. Leave `/scan` context-free.
4. **Instead, add a context-free integrity guard to `scan()`:** after loading `$fixture` and `$parent`, if `$fixture->app !== $parent->app` return **422** (`'This QR is not valid for this match.'`) before any scan row is written. This is what actually prevents a cross-app scan, and it holds regardless of headers. (App-one parent + app-one fixture â†’ passes; app-two member + app-two fixture â†’ passes.)
5. **Make app one explicit:** `demo_app_one/lib/app.dart` â†’ `AppConfig.fromEnvironment(defaultBaseUrl: demoOneApiBaseUrl, appKey: 'app_one')`. Harmless everywhere else (the header is only read where the middleware runs, and app one never calls `/content/*`). Leave `demo_app_three` alone (placeholder; its null appKey â†’ the `app_one` default â†’ today's behaviour).
