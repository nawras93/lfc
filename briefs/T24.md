Mobile-only — **no backend/PHP changes** (`php artisan test` stays green & untouched; every endpoint already exists from T21). Depends on **T21** (`/me/benefits`, `/offers` scoping, the `membership` block) + **T22/T23** (the supporter shell + role routing).

This fills the **Membership** tab for **`vvip_member`** accounts — replacing the T23 "coming soon" placeholder — with the premium VVIP experience the product calls for: a **digital membership card** (tier, member number, valid-until, identity QR, tier accent colour), the tier's **benefit catalogue**, and the member's **VVIP offers**. Everything is **showcase / non-transactional** (locked decision 5). Normal members keep the T23 discount wallet; guests keep the auth gate — this task only touches the `vvip_member` branch.

## Product shape (locked earlier)
- **Single role-driven authed shell.** `MembershipGateScreen`'s `account_type == 'vvip_member'` branch (currently a placeholder card) renders the new **`VvipCardScreen`**. The `member` (discount wallet) and guest (auth) branches stay **exactly as T23 built them**.
- **Non-transactional:** benefits + offers are display-only. No redemption/booking.
- **Card QR = static identity QR of `member_number`** (an entry/credential code), **NOT** the rotating `QrPassCard`. Rationale: `QrPassCard` fetches the **attendance/discount** scan token (`/scan-token`), which credits `member` accounts only — a `vvip_member` doesn't accrue discount, so reusing it here would be semantically wrong and misleading. Use a plain `QrImageView(data: memberNumber)` on the card.

## Non-negotiable: app one + the T23 wallet unchanged
Existing app-one behaviour + the T23 member-wallet flow stay green — especially the app-one `QrScreen`/`LoginScreen`/session tests and the T23 member wallet + auth tests. The **only** T23 test you may edit is the one that asserts the VVIP placeholder text (see Deliverable 5) — it is now obsolete because T24 replaces that placeholder.

## Deliverables

### 1. Membership feature in `lfc_core` (`lib/src/features/membership/`)
Consume the existing **`GET /me/benefits`** (T21) — it returns the whole card in one call, so no second round-trip for the card + benefits:
```json
{ "data": {
  "tier": { "name": "Platinum", "level": 2, "accent_color": "#C8A24A" },
  "member_number": "LSC-000123",
  "valid_until": "2027-06-30",
  "benefits": [ { "title": "...", "description": "...", "icon": "star" } ]
} }
```
…or `{ "data": null }` for an account with no tier.
- **Models** (plain immutable, `fromJson`): `MembershipBenefit` (title, description, icon — all `String?`), `MembershipTierInfo` (name, level `int`, accentColor `String?`), `MembershipCard` (tier `MembershipTierInfo`, memberNumber `String?`, validUntil `DateTime?` from `valid_until`, benefits `List<MembershipBenefit>`).
- **`MembershipRepository`** (constructed from `dioProvider`): `Future<MembershipCard?> fetchMembership()` → `GET /me/benefits`, unwrap `{data}`; return `null` when `data == null`, else map. Wrap in the existing `try/catch → ApiException.fromObject` pattern (mirror `AuthRepository.getMe`).
- Providers in `providers.dart`: `membershipRepositoryProvider` + `membershipProvider` (`FutureProvider.autoDispose<MembershipCard?>`), and an `offersProvider` (`FutureProvider.autoDispose<List<OfferSummary>>` → `offersRepositoryProvider.fetchOffers()`) for the offers section. (`offersRepositoryProvider` + `OfferSummary` already exist — reuse; the `OffersScreen` is app-one's, don't touch it.)

### 2. `VvipCardScreen` (the `vvip_member` view)
A scrollable, **pull-to-refresh** screen (single `ListView`/`CustomScrollView`, wide content never scrolls the body sideways) with three sections, driven by `membershipProvider` + `offersProvider` via the existing `.when(data/loading/error)` + `ApiException` pattern:
1. **Digital membership card** — a premium card using the tier **accent colour** (parse `accent_color` hex → `Color`; fall back to `context.lfc.gold` when null). Show: the brand crest/wordmark or tier name as the header, **tier name** + a `VvipPill` (reuse `theme/widgets/pills.dart`), the **`member_number`**, **"Valid until {date}"** (format with `AppDateFormat.date(locale)` → `dd-MM-yyyy`, see [[mobile-design-system]]), and the **static identity QR** (`QrImageView(data: memberNumber, size: ...)` on a white chip so it scans; key `Key('vvip-qr')`). Echo the app-one hero/broadcast-card treatment (gradient/navy-gold, `heroGradient`/`onHero`) but blended with the tier accent.
2. **Benefits** — section title ("Your benefits") + the `benefits` list: each row = the `icon` (map a small set of Heroicon-ish names/emoji to `IconData`, default `Icons.star_outline` for unknown/empty), `title`, `description`. Empty list → a friendly "No benefits listed yet" state.
3. **VVIP offers** — section title ("VVIP offers") + `offersProvider` list, **reusing the app-one offer-card look** (gold accent bar + `VvipPill` when `offer.audience == 'vvip'`, title, body, validity via `AppDateFormat`). Empty → reuse the `offersEmpty` copy. (Don't embed the whole `OffersScreen` widget — it's a self-contained scrollable; build a compact section list here instead.)
- If `membershipProvider` resolves to **`null`** (defensive — a `vvip_member` should always have a tier, but a mis-seeded one might not), show a graceful "Membership details unavailable" card rather than crashing.
- **Pull-to-refresh** invalidates `membershipProvider` + `offersProvider`.
