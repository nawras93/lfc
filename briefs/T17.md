# T17 — App-scoping foundation (shared app, data scoped by `app`)

**Branch:** `task/T17-app-scoping` (cut from current `main`; verify base with `git merge-base --is-ancestor main HEAD`). One task, one branch. **Commit everything before handoff** (clean worktree). **Run migrations + the full suite before committing** — never commit failing tests.

**This is backend-only (Laravel/PHP). No Filament panel changes, no API changes, no mobile changes, and NO app-two data yet.** This task lays the *spine* the whole Demo-App-Two epic scopes against: one Laravel app + one DB, with every per-app row tagged by an `app` column and a global scope that keeps app one and app two from ever seeing each other's data.

## Why
We're adding a second demo product (**app two = the club's fan/supporter app**) on the *same* Laravel app + DB as app one (the academy app). Rather than a second database or service, we scope data with an `app` discriminator. T18 will split `/admin` into per-app Filament panels that rely on this scope; T19–T21 add app-two features. **Get the scope right here and everything downstream is a filtered read.**

## Non-negotiable: app one must not change behaviour
The existing suite (**128 PHPUnit tests**) must stay **100% green** and `php artisan migrate:fresh --seed` must run clean. App one's data, queries, admin, API, and mobile all continue to work exactly as today. This task only *adds* a column (defaulted + backfilled to `app_one`), two enum cases, a ledger `unit` column, and a reusable scope trait.

## Deliverables

### 1. `App\Enums\AppKey`
A string-backed enum:
```php
enum AppKey: string
{
    case AppOne = 'app_one';   // academy app (today)
    case AppTwo = 'app_two';   // fan/supporter app (new)
}
```
Follow the existing enum conventions in `app/Enums/` (e.g. `AccountType`) — if the other enums implement `HasLabel` with `__('enums.app_keys.<name>')`, mirror that and add the keys to `lang/{en,ar}/enums.php` with EN/AR parity. (Labels only — this enum isn't user-facing yet, but keep the convention consistent.)

### 2. `app` column on the per-app tables
Add a `string('app')->default(AppKey::AppOne->value)->index()` column via migration to:
- `parent_accounts`
- `fixtures`
- `offers`

**Backfill:** because of the default, existing rows become `app_one` automatically on migrate. If you prefer explicitness, add an `->update(['app' => 'app_one'])` in the migration `up()` after adding the column — either way, **assert in a test that every pre-existing seeded row is `app_one`.** Add matching `$table->dropColumn('app')` in `down()`.

> Only these three tables get `app` in T17. `point_transactions` is scoped *transitively* through its owner account (owner = player XOR account, both ultimately app-scoped); `news_posts` / `standings` / `membership_*` don't exist yet (T20/T21 create them already `app`-scoped). Don't add `app` to candidate/team/season tables — those stay app-one-only for the demo (app two has no academy data of its own).

### 3. `App\Enums\AccountType` — two new cases
Extend the existing enum (currently `Parent`, `VvipClient`) with:
```php
case Member = 'member';           // app-two self-registered fan
case VvipMember = 'vvip_member';  // app-two admin-created premium member
```
Add their labels to `lang/{en,ar}/enums.php` (EN + accurate MSA Arabic, key parity). **Do not** wire any behaviour to them yet (T19/T21 do) — just define the cases so the column can hold them.

### 4. `unit` column on `point_transactions`
Add `string('unit', 20)->default('points')` (consider a small `App\Enums\LedgerUnit { Points='points'; DiscountPct='discount_pct'; }` for type-safety, matching the enum convention). This lets a member's balance read unambiguously as an accrued **discount %** (T19) rather than academy points, while app one's rows stay `points`. **Backfill existing rows to `points`** (the default handles it). The append-only invariant is untouched — `unit` is set at insert, never edited.

### 5. The scope trait — `App\Models\Concerns\ScopedToApp`
A reusable trait applied to `ParentAccount`, `Fixture`, `Offer` that:
- exposes the `app` attribute (cast to `AppKey` if you enum-cast; otherwise plain string is fine),
- adds a query scope `forApp(AppKey $app)` (or `->scopeForApp`),
- registers a **global scope** that filters to a resolvable "current app" **only when one is set**, and is a **no-op when unset** (so CLI/seeders/tests that don't set a context behave as today).

**Resolving "current app":** introduce a tiny context holder — e.g. an `App\Support\AppContext` singleton (bound in a service provider) with `current(): ?AppKey` / `setCurrent(AppKey)`. In T17 nothing sets it, so the global scope is inert and **app one behaves exactly as today**. T18's panels will call `AppContext::setCurrent(...)` per panel so each admin panel sees only its app's rows; the guest content API (T20) sets it from the `X-App-Key` header. Provide `Model::withoutAppScope()` (or rely on `withoutGlobalScope`) for the rare cross-app query.

> Design goal: **opt-in enforcement.** With no context set, queries are unscoped (today's behaviour, all tests green). With a context set, the model only returns that app's rows. Prove **both** directions in tests.
