## Fix 1 вЂ” Parent header "Account balance" card shows 0

**Where:** `home_shell.dart` вЂ” the `_InfoChip` with `balanceLabel` inside `_Body`.

**Why:** `account.accountBalance` is the **account-level** ledger (from `/me`'s `account_balance` = `ParentAccount::pointsBalance()`). A **parent's** points live on their **players** (candidate ledger), not the account, so a parent's account balance is always `0` вЂ” only a `vvip_client` has a meaningful account balance. Showing "Account balance: 0 pts" to a parent looks broken.

**Fix:** compute the header points chip by account type:
- `account.accountType == 'vvip_client'` в†’ show `account.accountBalance` with label `balanceLabel` ("Account balance") вЂ” unchanged behavior.
- otherwise (parent) в†’ show the **sum of the parent's players' `pointsBalance`** with label `totalPointsLabel` ("Total points" вЂ” **already exists** in both ARB files).

`HomeShell` is already a `ConsumerStatefulWidget`, so `ref.watch(playersProvider)` there, compute the total from its `AsyncValue` (`.maybeWhen(data: (players) => sum of pointsBalance, orElse: () => null)`), and pass the resolved value + label down into `_Body` (keep `_Body` presentational). While `playersProvider` is loading, show `вЂ¦` or the previous value вЂ” don't flash `0`.

Because the header now watches `playersProvider`, it will **auto-update after a redeem or a staff scan** (T10's Fix C already calls `ref.invalidate(playersProvider)` on both). Verify that still holds after this change.

> *Acceptable simpler fallback if the players-total wiring is awkward:* hide the balance chip entirely for parents and show it only for `vvip_client`. Prefer showing the players-total вЂ” it's more useful and demonstrates the live refresh.

## Fix 2 вЂ” "Insufficient points" shows the wrong redeem error

**Where:** `rewards_screen.dart` в†’ `_friendlyError(AppLocalizations, ApiException)`.

**Why:** the backend throws `InsufficientPointsException` as a **422** with message *"Insufficient points for this redemption."* (player path) or *"Insufficient account balance for this redemption."* (VVIP account path). `_friendlyError` has no branch for it, so it falls into the generic `422 в†’ redeemUnavailableError` ("This reward is unavailable right now.") вЂ” misleading.

**Fix:** add a branch **before** the generic 422 catch-all, matching the existing string-match style (it already matches `'linked'` and `'vvip client account'`):
```dart
if (error.statusCode == 422 && message.contains('insufficient')) {
  return l10n.redeemInsufficientError;
}
```
Add a new `redeemInsufficientError` key to **both** `mobile/lib/l10n/app_en.arb` and `app_ar.arb`:
- EN: `"You don't have enough points for this reward."`
- AR: `"Щ„Ш§ ШЄЩ…Щ„Щѓ Щ†Щ‚Ш§Ш·Щ‹Ш§ ЩѓШ§ЩЃЩЉШ© Щ„Щ‡Ш°Щ‡ Ш§Щ„Щ…ЩѓШ§ЩЃШЈШ©."`

Run `flutter gen-l10n`. (Existing redeem keys for reference: `redeemLinkedPlayerError`, `redeemVvipOnlyError`, `redeemUnavailableError`.)

## Fix 3 вЂ” Hide the VVIP status chip for non-VVIP accounts

**Where:** `home_shell.dart` вЂ” the `_InfoChip` with `vvipStatusLabel` in `_Body`.

**Why:** it currently renders "VVIP: No" for every parent, which is meaningless noise.

**Fix:** only render the chip when the account is VVIP:
```dart
if (account.isVvip)
  _InfoChip(label: l10n.vvipStatusLabel, value: l10n.yesText),
```
Since it only shows when `isVvip` is true, the value is always "Yes" вЂ” drop the `isVvip ? yesText : noText` ternary. No new strings.

> **Net result of Fixes 1 + 3:** a plain parent's header reads **Account type + Total points**; a VVIP's reads **Account type + VVIP + Account balance**. Confirm both look right in EN **and** AR/RTL.
