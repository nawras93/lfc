## Deliverable 1 — schema

**`membership_tiers`** (new): `app` (`string` default `AppKey::AppOne->value`, indexed — T17 convention), `name` + `name_ar` (nullable), `level` (`unsignedInteger` — higher = more premium; drives ordering), `accent_color` (`string` nullable — hex for the card, e.g. `#C8A24A`), `is_active` (`bool` default true), `timestamps`.

**`membership_benefits`** (new): `membership_tier_id` (FK → `membership_tiers`, `cascadeOnDelete`), `title` + `title_ar` (nullable), `description` + `description_ar` (nullable), `icon` (`string` nullable — a Heroicon name or emoji), `sort_order` (`unsignedInteger` default 0), `timestamps`. (No `app` column — reached via its tier, which is app-scoped.)

**`parent_accounts`** (extend, migration): `membership_tier_id` (`nullable` FK → `membership_tiers`, `nullOnDelete`), `member_number` (`string` nullable), `membership_valid_until` (`date` nullable). These are only populated for `vvip_member` accounts; app-one/other accounts leave them null (no behaviour change).

## Deliverable 3 — API

### `GET /api/v1/me/benefits` (auth, new)
Guard `instanceof ParentAccount` (403 otherwise, like `OfferController`). Return the caller's membership as localised JSON:
```json
{ "data": {
    "tier": { "name": "...", "level": 2, "accent_color": "#C8A24A" },
    "member_number": "LSC-000123",
    "valid_until": "2027-06-30",
    "benefits": [ { "title": "...", "description": "...", "icon": "..." } ]
} }
```
If the account has no tier (normal member / app-one account), return `{ "data": null }` (200). Route it in the `auth:sanctum` group.

### `/me` (extend `ParentAccountResource`)
Add a compact `membership` block so the client can render the card without a second call: `membership => $this->membership_tier_id ? { tier_name: localized, tier_level, member_number, valid_until } : null`. Keep the existing fields (incl. T19's `discount_percent`) intact.

### `GET /offers` — **fix the cross-app leak** (important)
`OfferController::index` currently does `Offer::query()->visibleTo($parent)` with **no app scoping** — harmless until now, but once app-two VVIP offers exist it would serve **app-one offers to app-two members and vice-versa** (the global scope is inert on API requests). Fix: `Offer::query()->forApp($parent->app)->visibleTo($parent)`. `$parent->app` is the cast `AppKey`; `forApp` is a plain filter here (global scope inert). This keeps app one identical (its accounts are `app_one`, all offers `app_one`) and correctly isolates app two. **Add a test** (below).
