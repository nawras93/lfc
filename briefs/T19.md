## Key implementation decision — accrual is a config constant, NOT a PointRule

The T18 backlog row says "fixed PointRule scoped to app_two." **Don't use the `point_rules` engine.** `PointsEngine::resolveRule()` resolves by fixture **team/season** (the academy model) and has no `app` concept; shoehorning app-two through it would mean adding `app` to `point_rules` and rewriting `resolveRule()` — needless risk to app one. Instead define the accrual + cap as **config constants** (`config/loyalty.php`). Admin-editable accrual rate is a possible later enhancement, out of scope here.

### The unit problem (read carefully)
`point_transactions.points` is an **integer** column — it can't hold `0.5`. Store the discount in **hundredths of a percent** ("percent basis points", bp): **`+50` per attendance = 0.5%**, cap **`1000` = 10%**. The ledger row carries `unit = discount_pct` (the T17 `LedgerUnit` enum) so it's never confused with app-one points. The server converts to a human percentage on the way out: `discount_percent = discountBalance / 100`.

```php
// config/loyalty.php
return [
    'app_two' => [
        'discount_bp_per_attendance' => 50,   // 0.5%
        'discount_cap_bp' => 1000,            // 10%
    ],
];
```

- Public route (outside `auth:sanctum`, inside the existing `v1` group so `throttle:api` + `api.locale` apply). Add it in `routes/api.php` beside `/auth/login`.
- Validate `name` (required), `email` (`required,email,unique:parent_accounts,email`), `password` (`required,string,min:8,confirmed`), `phone` (nullable).
- **Create the account with `app` set explicitly** (this is the first exercise of the T17 write-path rule — `ScopedToApp` does NOT auto-stamp): `app => AppKey::AppTwo`, `account_type => AccountType::Member`, `is_vvip => false`, `accepted_at => now()` (self-registered, no invite). Password is hashed by the model's `'password' => 'hashed'` cast — just pass the plain value.
- App one stays invite-only — do **not** touch `acceptInvite()`.
- Note (acceptable for demo): `email` has a table-wide unique index, so uniqueness is global across app one + app two. Fine; do not add a migration to scope it per-app.

### 2. `ParentAccount` model — separate the two balances
`pointsBalance()` currently sums **all** the account's transactions. A member only has `discount_pct` rows, so left as-is `account_balance` would leak the bp sub-unit. Fix:
- `pointsBalance()` → sum only `where('unit', LedgerUnit::Points->value)` (app-one accounts are all `points`, so **no change for them**).
- add `discountBalance(): int` → sum only `where('unit', LedgerUnit::DiscountPct->value)`.
- add helpers `isMember(): bool` (`account_type === AccountType::Member`) and `discountPercent(): float` (`discountBalance() / 100`).

### 3. `PointsEngine` — one new method (don't touch the existing ones)
```php
public function creditAttendanceDiscount(ParentAccount $account, int $bp, Model $source): PointTransaction
{
    return PointTransaction::query()->create([
        'parent_account_id' => $account->id,
        'type' => PointTransactionType::Earn,
        'points' => $bp,                       // positive; hundredths-of-a-percent
        'unit' => LedgerUnit::DiscountPct,
        'source_type' => $source->getMorphClass(),
        'source_id' => $source->getKey(),
    ]);
}
```
Append-only guards + XOR-owner (account-owned here) invariants are unchanged.

### 4. `ScanController::scan()` — branch on account type (the core change)
After `$parent = ParentAccount::query()->findOrFail($parentId)` (global scope is inert on API requests, so it resolves members fine), branch **before** the linked-players logic:

- **If `$parent->isMember()`** (app-two member):
  - Compute the capped accrual: `$cap = config('loyalty.app_two.discount_cap_bp'); $per = config('loyalty.app_two.discount_bp_per_attendance'); $add = max(0, min($per, $cap - $parent->discountBalance()));`
  - In a `DB::transaction`: create the `AttendanceScan` (same `parent_account_id`+`fixture_id`+`scanned_by`+`scanned_at`, so the existing `unique(parent_account_id, fixture_id)` dedupe still fires), then **if `$add > 0`** call `creditAttendanceDiscount($parent, $add, $scan)`. If `$add === 0` (cap reached) still record the scan, credit nothing.
  - Reuse the existing `catch (UniqueConstraintViolationException)` → `409 'Already scanned for this match.'`
  - Return a member-shaped payload, e.g. `{ scan_id, discount_added_percent: $add/100, discount_percent: $parent->fresh()->discountPercent(), discount_cap_percent: $cap/100 }`.
- **Else** (existing parent path): the current linked-players credit loop, **verbatim** — including the "No linked player on this match's team" 422. Do not alter it.

> The accrual is per-attendance and does **not** depend on the fixture's team or app — a member simply earns 0.5% for being scanned into any open fixture. (App-two fixtures arrive in T20; T19 tests can scan a member into any open fixture.) Keep the branch purely account-type-driven.

The member's QR still comes from the existing `GET /scan-token` (a member **is** a `ParentAccount`, so `token()`'s `instanceof ParentAccount` check already passes) and the staff scanner still drives `POST /scan`. No changes needed there beyond the branch above.
