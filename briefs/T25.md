## 🚨 Two non-negotiables (read first)

1. **`DatabaseSeeder` uses `WithoutModelEvents` → the `ScopedToApp` auto-stamp does NOT fire during seeding.** Model events are muted for the whole seed run (including sub-seeders called via `->call()`). So the T20 "auto-stamp `app` on create" **will not run here** — **every app-scoped row you insert MUST set `app` EXPLICITLY** (`'app' => \App\Enums\AppKey::AppTwo->value`). App-scoped tables: `parent_accounts`, `fixtures`, `offers`, `news_posts`, `standings`, `membership_tiers`. (`membership_benefits`, `point_transactions`, `attendance_scans` have **no** `app` column — they're reached through an app-scoped parent, so leave them as-is.) A row you forget to stamp silently becomes `app_one` and **leaks into app one's panel + API** — this is the single biggest footgun in this task. Do not rely on setting `AppContext` instead; the muted `creating` event means it won't be read.
2. **AUTHZ invariant still holds:** do **NOT** run `php artisan shield:generate` and do **NOT** add any model policy. T25 adds no resources, but keep this in mind if you touch anything Filament-adjacent. Only `RolePolicy` may exist.

Also: **`parent_accounts.email` is globally unique** (not per-app). The new app-two accounts MUST use emails distinct from the app-one demo accounts (`parent.demo@`, `parent2.demo@`, `vvip.demo@`). And **app-two `offers` titles must be distinct** from the app-one seeded offer titles (`Early Bird Registration Discount`, `VVIP Lounge Access — Al Thumama Match`, `Parent Workshop…`) so the existing app-one offers invariant assertions stay valid.
