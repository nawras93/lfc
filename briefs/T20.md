## Architecture decision — extend `fixtures`, don't make a new match table

App-two "matches" (first-team fixtures vs external clubs, with scores) are **the same rows** a member gets scanned into for their attendance discount — **T19 already wired app-two attendance to `fixtures`** (`AttendanceScan.fixture_id → fixtures`). Splitting content into a separate `club_matches` table would divorce "the match a fan browses" from "the match they're scanned into." So **extend `fixtures`** with the public-match fields. News and standings are genuinely new concepts → new tables.

`fixtures.team_id` is currently **required** (academy team). App-two matches have no academy team, so make it **nullable** (app-one always sets it, so this is backward-compatible). Verify `Fixture::isOpenForScanning()` and the app-one scan path don't dereference `team_id` in a way that breaks on null (the app-two member scan branch from T19 already skips the team/player logic).

## Deliverable 0 (foundation) — `ScopedToApp` auto-stamps `app` on create

T19's lesson: `ScopedToApp` scopes **reads** but not **writes**, so every app-two create must remember to set `app`. Fix it once, centrally. In the trait's `bootScopedToApp()`, add a `creating` hook:

```php
static::creating(function ($model): void {
    if ($model->getAttribute('app') === null && ($app = app(AppContext::class)->current()) !== null) {
        $model->setAttribute('app', $app);
    }
});
```

**Why this is safe for app one:** it only fires when an `AppContext` is set — i.e. inside a panel request (T18 middleware) or the guest-API group (below). The app-one panel sets `AppContext=app_one`, so app-one Filament creates get stamped `app_one` — identical to the existing DB default (no behaviour change). In CLI/seeders/most existing tests no context is set → no stamping → unchanged. Explicit values still win (e.g. T19 registration sets `app_two` outside any context). **Add a test:** with `AppContext::setCurrent(AppKey::AppTwo)`, a newly-created `Offer`/`NewsPost` (no `app` passed) persists as `app_two`; with no context, it falls to the column default.

> Net effect: every app-two admin resource in this task can create rows **without** manually stamping `app` — the panel context does it. (Seeders in T25 run context-less, so they'll still stamp explicitly.)

## Deliverable 4 — guest content API (no auth)

Add a header-driven context middleware `App\Http\Middleware\SetAppContextFromHeader`: read `X-App-Key`; if it's a valid `AppKey` use it, else default `AppKey::AppTwo`; `app(AppContext::class)->setCurrent(...)`. (These are app-two content endpoints; the default keeps them working if the header is absent. Future app_three would send its own key.)

In `routes/api.php`, **inside** the `v1` group (so `throttle:api` + `api.locale` apply) but **outside** `auth:sanctum`, add a group wrapped in `SetAppContextFromHeader`:
```
GET /content/news            → NewsController@index   (published only, list: id,title,excerpt,image_url,published_at)
GET /content/news/{news}     → NewsController@show    (published only, full body; 404 if unpublished)
GET /content/fixtures        → MatchController@fixtures (upcoming: not played, kickoff future, asc)
GET /content/results         → MatchController@results  (played: both scores set, kickoff desc)
GET /content/standings       → StandingController@index (ordered; include derived position + goal_difference)
```
Controllers query the models normally — `ScopedToApp`'s global scope (now that the middleware set the context) filters to `app_two`. Return `{ data: [...] }` with **localised** free-text via `->localized('title')` etc. (the `api.locale` middleware already sets the locale from `Accept-Language`/`?lang`), mirroring `OfferController`. `image_url` = `$post->image_path ? Storage::url($post->image_path) : null`.
