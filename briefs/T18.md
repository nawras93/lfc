## Why
We're standing up a second demo product (app two = the fan/supporter app) on the same Laravel app. The admin needs to become **two panels** — `/admin-app-one` (today's academy admin, unchanged) and `/admin-app-two` (the fan-app admin, empty shell for now) — each wired to T17's `AppContext` so its data is auto-scoped to its own app. This task does the split + rename + per-panel scoping wiring, so T19–T21 can just drop resources into the app-two panel.

## Non-negotiable: app one stays identical
- The existing **169-test suite** stays green; `migrate:fresh --seed` clean.
- App-one panel = today's panel with a new id/path only. **Same resources, widgets, branding, nav, permissions, dark-mode default.** Do **not** move or rename the existing `App\Filament\Resources\*` / `App\Filament\Widgets\*` classes (see the architecture decision below — moving them is out of scope and risky).

## Architecture decision — DO NOT move app-one's resources (low-churn split)

The obvious-but-wrong approach is to re-home everything into symmetric `App\Filament\AppOne\**` + `App\Filament\AppTwo\**`. **Don't.** Moving app one's ~40 resource/page/schema/table/relation-manager files means: rewriting every namespace, updating every `Livewire::test(...\Pages\...)` import in the resource tests, and — critically — **Filament Shield derives permission names from resource FQNs**, so a namespace move can rename/regenerate permissions and perturb the seeded roles. For a task whose whole point is *app one unchanged*, that's unacceptable churn.

**Instead — asymmetric, minimal:**
- App-one panel keeps discovering its **existing** `App\Filament\Resources` + `App\Filament\Widgets` (the implicit "app one" set — untouched).
- App-two panel discovers a **new, initially-empty** `App\Filament\AppTwo\Resources` + `App\Filament\AppTwo\Widgets`.
- The two panels discover **disjoint directories**, so neither leaks the other's resources — which is the only property we actually need. (Filament auto-discovery is per-panel; a panel only shows what *it* discovers/registers.)

Accept the naming asymmetry (`Resources` = app one, `AppTwo\Resources` = app two); add a one-line comment in each panel provider noting it. This keeps the app-one diff to essentially "id + path + one middleware."

### 4. `App\Http\Middleware\SetAppContext` (new) — the per-panel scoping seam
A parametrized middleware that sets T17's request-scoped app context so every `ScopedToApp` query in that panel is auto-filtered:
```php
public function handle(Request $request, Closure $next, string $app): Response
{
    app(AppContext::class)->setCurrent(AppKey::from($app));
    return $next($request);
}
```
Registered per panel as shown in #2/#3. Effect: on `/admin-app-one/*` requests every `ParentAccount`/`Fixture`/`Offer` query gains `where app = 'app_one'` (all existing rows are `app_one`, so **no behaviour change today**, but the app-one admin is automatically isolated from app-two data once it exists); `/admin-app-two/*` filters to `app_two`.
> **Per-request note (from the T17 review):** `AppContext` is a container singleton — request-scoped under `php artisan serve` (fine for the demo). This middleware sets it each request; don't cache it statically. Under Octane you'd also reset it, but that's out of scope here.
> **Write-path note (T17 heads-up #1):** this scopes **reads**, not writes. App-one Filament creates rely on the `app` DB default (`app_one`) — fine, no change. App-two create forms (T19+) must set `app_two` explicitly; not this task's concern, just don't add auto-stamping here.
